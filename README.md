# Kogu

> A dictionary for the living Han-script written systems (Traditional Chinese, Simplified
> Chinese, Cantonese, Japanese) whose headline feature is explaining *why* the same idea is
> written or read differently across them.

**Live at <https://kogu.miro.build>.**

## What it is

A free, server-backed online dictionary with three capabilities most CJK dictionaries lack:

1. **A "why it branched" engine** (lexical, orthographic, phonological), built only from
   open datasets: no LLM, no generated prose.
2. **Any-to-any handwriting input**: draw a character your keyboard can't type, in any of
   the four systems, and get candidates shown across all of them.
3. **English-pivot translation search** via a language-independent concept layer.

## Scope

In: Traditional Chinese (TW/HK/Macau), Simplified Chinese (mainland/SG), Cantonese
(first-class, incl. 粵字 + jyutping), Japanese (shinjitai/kyūjitai). Region is a first-class tag.
Out (deliberately): Korean, Vietnamese. Offline use is deferred (see design doc, Appendix A).

## Architecture

| Layer            | Choice                                                                    |
|------------------|---------------------------------------------------------------------------|
| Backend (serving)| **Rust**: Axum + rusqlite (SQLite/FTS5), single static binary             |
| Data store       | One precompiled **SQLite** file (FTS5), memory-resident at runtime        |
| Build pipeline   | Offline ingestion (Python/JS); **no Python in the serving path**          |
| Handwriting      | Proxy of the **Google Input Tools** endpoint                              |
| OCR              | **Local** PP-OCRv5 (PaddleOCR mobile) ONNX models via `ort`, in-process   |
| Japanese TTS     | Local **OpenJTalk** sidecar, forcing **Kanjium** pitch accents            |
| zh/yue audio     | Per-syllable mp3 clips proxied from third-party sources (see `NOTICE.md`) |
| `/mt` translate  | Proxy of Google Translate's unofficial `gtx` endpoint, MyMemory fallback  |
| Frontend         | **Svelte + Vite** SPA/PWA; `<canvas>` ink to `/recognize`                 |
| Reverse proxy    | nginx serving the static frontend, proxying `/api` to the backend binary  |

## Layout

```
kogu/
├── backend/    Rust serving binary (Axum + rusqlite)
├── frontend/   Svelte + Vite SPA/PWA
├── pipeline/   Offline ingestion: open sources -> one normalised SQLite DB
├── tts/        Japanese TTS sidecar (OpenJTalk, loopback-only)
├── deploy/     deploy script, systemd unit, nginx vhost
├── bin/        Installed release binary (gitignored)
└── data/       Built + downloaded artifacts (gitignored, regenerated by pipeline)
```

## Status

Live and feature-complete through phase 3 of the build plan: the "why" engine (lexical
etymology + orthographic reforms + phonological readings incl. Middle Chinese), the concept
layer (translate + cognate/false-friend), first-class Cantonese, region tags, handwriting,
OCR, and audio.

## Building

- **Backend**: `cd backend && cargo build --release`. Run tests with `cargo test --release`
  (the OCR dependencies are very large in the dev profile).
- **Frontend**: `cd frontend && pnpm install && pnpm build`.
- **Database**: see `pipeline/README.md` for the fetch + build order. The backend expects
  `data/kogu.sqlite` (override with `KOGU_DB`).
